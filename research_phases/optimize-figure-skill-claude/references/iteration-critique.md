# 多轮迭代优化（Iteration Critique）

本文件定义第 2 轮起的"看图反馈 -> 修订 prompt"流程。这是本 Skill 相对 `happy-figure-skill-main` 的核心增量能力：不生成图片，但会基于**用户提供的已生成图片 + 本轮修改要求**，对上一轮 prompt 做结构化诊断和最小改动修订。

## 何时进入本流程

当用户在同一次对话中，针对**已经输出过的绘图提示词**再次发言，并且符合以下任一模式：

- 上传/引用了"这版 prompt 生成出来的图片"，并提出修改意见（例如"箭头方向反了""模块间距太挤""字体太多"）。
- 没有上传图片，但用文字描述了对已生成图片的不满（例如"图2 的中心模块不够突出""标签太密"）。
- 明确说"这是第X版的结果，我要优化"或类似表述。

不要把"用户提出一个全新的绘图需求"误判为迭代——迭代的前提是**存在一个当前对话内已经生成过的 prompt 版本（Figure Brief + 模型 + 完整 prompt）**可以作为修订基准。如果没有这个基准（例如新会话、或用户明显在问一张不相关的图），走 `SKILL.md` 的常规首轮流程。

## 单会话范围

本流程只在**当前对话上下文**内工作，不做跨会话持久化：

- 迭代所需的状态——Figure Brief（图像目标/领域对象/图类型结构/区域-连接关系/可见文字清单/科学边界）、已确认的模型（Nano Banana Pro / Nano Banana 2 / gpt-image-2）、上一轮完整 prompt 文本——全部依赖对话历史本身，不写入任何本地文件。
- 如果用户希望换一个新对话继续迭代，需要重新粘贴上一版 prompt 和反馈意见，作为新的迭代基准起点。

## 读图能力判断

- 如果当前宿主模型具备读图能力，且用户上传了图片，直接分析图片；不要跳过看图直接凭文字反馈猜测问题。
- 如果宿主模型不能读图，或用户没有上传图片，要求用户用文字描述图片的具体问题（结构/标签/风格三个维度任选），不要假装看过图片。
- 如果图片和文字反馈都提供了，两者一起作为诊断输入，文字反馈优先于从图片推断出的问题（用户的主观判断优先于模型的客观读图）。

## 结构化诊断维度

参考顶级会议 Critic/审稿人的挑错逻辑，诊断时对照本轮生成图片和该版本的 Figure Brief，逐项检查（不需要每项都输出，只输出发现的问题）：

1. **结构保真度**：图片呈现的区域数量、模块划分、箭头方向/连接关系是否与 Figure Brief 一致；是否有区域被合并、遗漏或顺序错乱。
2. **领域语义一致性**：领域母版规定的视觉语言（如 CS 的 Content Community + Visual Treatment、材料学的晶格/球棍模型、生物医学的细胞/信号通路符号）是否被正确呈现，有没有漂移成通用图标风或错误学科视觉语言。
3. **可见文字/标签准确性**：图中出现的文字是否只包含 Figure Brief 可见文字清单里的内容；有没有拼写错误、乱码、清单外的多余文字、或该有的标签缺失。
4. **科学边界**：有没有编造 Figure Brief 明确排除的数据、曲线、公式、实验结果或设备细节。
5. **模型特定常见失败**（按当前确认模型选择对应检查项）：
   - Nano Banana Pro：区域拥挤、跨区域标签串位、复杂结构下箭头语义模糊。
   - gpt-image-2：文字渲染正确但说明性短语过多导致画面拥挤、视觉焦点被文字块盖过。
   - Nano Banana 2：标签渲染失真/截断、结构简化过度丢失关键模块（提醒用户这是草稿模型的已知局限，而非 prompt 问题）。
6. **用户本轮提出的具体问题**：无论是否落在上述 1-5 类，都必须逐条对应到修订动作，不能遗漏。

## 修订原则（优先级）

修订时严格按以下优先级排序，与首轮生成的优先级基本一致，但把"本轮用户反馈"提到最高：

```
科学事实 > 用户本轮明确反馈 > 原 Figure Brief 结构 > 模型标签密度规则（model-cards.md / language-strategy.md）> 审美词
```

**核心约束：最小改动修订，不重写。** 修订版 prompt 应当是上一版 prompt 的局部编辑结果：
- 用户反馈涉及的区域/标签/风格描述做针对性修改。
- 用户未提及的区域、连接关系、可见文字清单、语言策略、模型适配语句保持原样，不因为"顺便优化一下"而改写。
- 如果原 prompt 某处确实存在明显问题、且与本轮反馈相关，才允许连带修正；不相关的部分不要动。
- 这与 `llmresearch-paperbanana-main` 的 Critic agent 采用同一约束：模糊或大幅重写的修订往往让图变差而不是变好。

## 何时需要打断迭代 / 先确认

以下情况不要直接生成修订版 prompt，先向用户确认：

- 反馈内容暗示要改变**图类型或领域判断本身**（例如从"技术路线图"改成"机制解释图"，或从 CS 领域改判为材料学），这超出"迭代修订"范围，属于要不要重新走首轮流程的问题。
- 用户要求切换生成模型（例如从 Nano Banana 2 换到 Nano Banana Pro 定稿）——此时应说明这是"基于同一 Figure Brief 为新模型重新生成 prompt 并重置轮次计数"，不是对上一版做修订。
- 反馈本身模糊、无法定位到具体区域或元素（例如"感觉不太好看"），先追问是结构、标签、色彩、构图哪个维度的问题。

## 循环终止

用户明确表示满意（如"可以了""就这版""满意"）即结束迭代，不需要额外的确认步骤或质检输出。

## 输出格式

```markdown
## 变更摘要（第 N 轮）
[3-6 条：本轮反馈的问题定位、对应的修改动作；哪些部分保持不变。]

## 优化后的绘图提示词（第 N 轮 · [模型名]）
[完整可复制 prompt，基于上一版最小改动修订而来。]

## 简短审查提醒
[沿用 prompt-quality-check.md 的一句话提醒，可省略如果上一轮已经提示过且本轮无新增风险点。]
```

如果用户明确说"只要 prompt"，只输出 `## 优化后的绘图提示词（第 N 轮 · [模型名]）`。
